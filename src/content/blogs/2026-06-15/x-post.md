Fable 5 changed the unit of AI work.

Not from bad output to good output.

From response to run.

That is the shift I cannot stop thinking about.

In the normal chat loop, I am inside the work.

I ask.
Inspect.
Correct.
Add context.
Ask again.

My attention is the control loop.

But when I gave Fable 5 a multi-hour task, the shape changed. It planned, delegated, researched, synthesized, and came back with polished pages.

The visible output was the page.

The more important product was the run.

A response is small. If it is wrong, I correct it in the next turn.

A run is different.

A run has duration.
State.
Tool access.
Cost.
Subagents.
Intermediate assumptions.
Progress claims.
Artifacts.

The cost of a bad assumption is no longer one bad answer. It can become an entire wrong branch of work.

That is why "prompt engineering" is not the right frame for long-running agents.

The scarce skill is run design.

Before the run:

What is the objective?
What context is authoritative?
What can the agent read or write?
Where should it pause?
What counts as evidence?
What is the budget?
How do I roll it back?

After the run:

What did it actually do?
What assumptions did it carry forward?
Which artifacts can I inspect?
Where did judgment still need to be human?

This is also why Superpowers clicked for me.

Superpowers is not another model. It is an open-source skills framework and software development methodology for coding agents.

It wraps the model in an operating discipline:

Clarify the goal.
Tease out a spec.
Turn the spec into a plan.
Execute through phases or subagents.
Review the work.
Verify before claiming completion.

The model is the engine.

The skills are the operating discipline around the engine.

That is the practical rehearsal for the Fable-style future.

The more autonomous agents become, the more important the surrounding workflow becomes.

This is not just project management with AI branding.

Agents move at machine speed.
They use tools.
They can take actions.
They are probabilistic workers without native accountability.

So the operating contract matters.

What can it do?
How will I know what it did?
How do I stop, reverse, or repair it if it goes wrong?

That is the new control surface.

The strongest model still matters.

But the strongest model inside a weak operating system can produce expensive chaos.

A slightly weaker model inside a disciplined workflow may produce shippable work.

The future does not belong to the best prompter.

It belongs to the builder who can let execution run ahead without letting judgment fall behind.
