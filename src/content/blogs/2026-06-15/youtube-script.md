# Fable 5 Changed the Unit of AI Work

## [HOOK: 00-cover.png]

Fable 5 did not just feel like a better model.

It felt like a different unit of work.

The old AI loop is simple:

Prompt.

Response.

Correction.

Another prompt.

But with Fable, I gave it a multi-hour task.

It planned, delegated, researched, synthesized, and came back with polished pages.

The visible output was the page.

The more important product was the run.

## [SLIDE: Response vs Run - 01-response-vs-run.png]

A response is small.

If it is wrong, I correct it in the next turn.

A run is different.

A run has duration.

State.

Tool access.

Cost.

Subagents.

Intermediate assumptions.

Artifacts.

The cost of a bad assumption is no longer one bad answer.

It can become an entire wrong branch of work.

## [SLIDE: Attention Moves - 02-attention-moves.png]

In prompt mode, my attention is the control loop.

I ask.

Inspect.

Correct.

Add context.

Ask again.

In run mode, my attention moves before and after the work.

Before the run, I define the conditions.

After the run, I inspect the evidence.

## [SLIDE: Run Design - 03-run-design.png]

That is why the scarce skill is not just prompting.

It is run design.

What is the objective?

What context is authoritative?

What can the agent read or write?

Where should it pause?

What evidence supports progress claims?

What is the budget?

How do I roll it back?

## [SLIDE: Superpowers Is Discipline - 04-superpowers-discipline.png]

This is why Superpowers clicked for me.

Superpowers is not another model.

It is an open-source skills framework and software development methodology for coding agents.

It wraps the model in a disciplined workflow:

Clarify the goal.

Write the spec.

Create the plan.

Execute through phases or subagents.

Review the work.

Verify before claiming completion.

## [SLIDE: Engine And Operating Discipline - 05-engine-discipline.png]

The model is the engine.

The skills are the operating discipline around the engine.

That is the practical rehearsal for the Fable-style future.

Better models make the surrounding workflow more important, not less.

The stronger the agent, the more the run needs boundaries.

## [SLIDE: Not Just Project Management - 06-not-just-project-management.png]

This is not just project management with AI branding.

Agents move at machine speed.

They use tools.

They can take actions.

They are probabilistic workers without native accountability.

So the operating contract matters.

What can it do?

How will I know what it did?

How do I stop or repair it?

## [SLIDE: Governance Is The Same Pattern - 07-governance-pattern.png]

Governance is the same problem at larger scale.

The risk is not intelligence by itself.

It is autonomy plus access without proportional control.

An agent that only observes is one category.

An agent that drafts is another.

An agent that acts with approval is another.

An agent that acts autonomously across systems is different again.

## [SLIDE: Operating Contract Stack - 08-operating-contract-stack.png]

For every serious agent run, I want an operating contract:

Objective.

Context.

Authority.

Checkpoints.

Evidence.

Budget.

Rollback.

This is how autonomy becomes usable.

## [SLIDE: The New Operator Skill - 09-new-operator-skill.png]

The strongest model still matters.

But the strongest model inside a weak operating system can produce expensive chaos.

A slightly weaker model inside a disciplined workflow may produce shippable work.

The future does not belong to the best prompter.

It belongs to the builder who can let execution run ahead without letting judgment fall behind.
