# FDE 文章工作流复盘：从“会生成”到“能出版”

## 结论先行

这篇文章真正验证的，不是 AI 能不能写出流畅文字，而是高质量内容需要一套持续参与、分阶段判断、能够把反馈写回系统的生产方式。

最终质量的提升，主要来自五件事：

1. 从一个宽泛话题找到真正值得写的矛盾；
2. 把事实、推断、个人判断和个人经验分开；
3. 用具体机制代替抽象概念堆叠；
4. 让每轮反馈改变系统，而不只修正一句话；
5. 把正文、图片和发布页面分别验收，再检查它们是否仍然一致。

换句话说，模型负责扩大探索和执行能力，人负责决定什么值得保留、什么证据足够、什么表达准确，以及哪些修正应该成为下一次运行的默认规则。

## 这篇文章经历了什么

文章最初从 private equity 和 AI 的关系开始。这个方向和 Mawer 的产品发布有关，但它很快暴露出两个问题：我的金融专业背景不足，而且如果围绕 Mawer 展开，文章会更像公司内容，而不是个人博客。

进一步研究后，真正有传播力的信号浮现出来：Anthropic、OpenAI、AWS、Microsoft 都在把工程师推向客户现场。Private equity 因此退回到背景，Forward Deployed Engineering 成为主角。这个变化不是换了一个标题，而是重新定义了文章要回答的问题：模型已经很强，为什么企业仍然需要大量工程师帮助它落地？

之后的主要修改可以归纳为七次重心校准：

| 阶段 | 当时的问题 | 最终修正 | 可复用的教训 |
|---|---|---|---|
| PE 与 AI | 权威边界不够稳，个人博客容易被公司叙事带走 | PE 只保留为触发点，文章转向 FDE | 选题要落在自己能够形成判断的交叉地带 |
| 四家公司新闻 | 容易写成公告摘要和公司列表 | 提炼成共同的 deployment signal | 来源提供事实，文章必须提供新的解释 |
| FDE 定义 | “human backpropagation”听起来正确但抽象 | 用制造商供应中断解释现场工作 | 机制和现实决策比概念分类更有说服力 |
| 个人博客工作流 | 一度被放在开头，无法支撑企业级主张 | 移到机制之后，明确只是缩小版例子 | 个人经历可以解释机制，不能越级充当市场证据 |
| ACTOR 与其他清单 | 多套框架争夺读者注意力 | 删除“四项产出”，只保留 ACTOR | 一篇严肃文章只保留一个主要执行框架 |
| 中英文表达 | 中英混杂、抽象词误用、频繁一句一段 | 分别审校，按完整观点组织段落 | 中文是独立改写，不是英文结构的逐句复制 |
| 配图和发布 | 风格重复、图意错误、正文改后媒体过期、链接曾失效 | 概念先行、候选比较、语义审图、本地浏览器验收 | 文件生成成功不等于读者拿到完整成品 |

## 最重要的质量跃迁

### 1. 从“找到资料”升级到“建立论证”

`research-dossier.md` 的价值是扩大事实边界，`claim-ledger.md` 的价值是限制表达边界，`argument-memo.md` 的价值是决定文章究竟要证明什么。三者分工清楚以后，资料不再按搜索顺序进入正文。

这次最典型的例子是 75 亿美元。数字可以形成强烈信号，但 OpenAI、AWS、Microsoft 的投入并不是同一种会计口径。Claim ledger 迫使文章把它写成规模信号，而不是精确可比的支出，更不能把公告当作客户 ROI 的证明。

以后每个重要主张都应该能够回答：

- 这是事实、推断、判断，还是个人观察？
- 一手来源真正支持到了哪一步？
- 读者最可能质疑什么？
- 删除这个数字或案例以后，论证是否仍然成立？

### 2. 从“解释概念”升级到“展示机制”

早期版本试图用 customer、operating、product 等层次解释 FDE，结构完整，却没有让读者看见工程师到底在做什么。制造商供应中断的例子更有效，因为它同时呈现了数据分散、权限、审批、异常、行动和产品反馈。

今后的机制段落应尽量包含：

```text
真实对象 -> 真实约束 -> 需要改变的决策或行动
-> 系统如何介入 -> 哪里会失败 -> 失败如何改进下一次运行
```

如果一段话只能说明“这个概念很重要”，却不能说明它如何工作，它还没有写完。

### 3. 从“个人故事优先”升级到“证据权重优先”

个人经验能提供人味和原创性，但不应该自动占据开头。FDE 文章存在的理由是四家公司的组织动作，因此文章必须先让读者看见这个市场信号。博客工作流适合在后面说明：真正打通并维护一套 AI 工作系统，需要持续判断和写回，而不是几句 prompt。

以后的开头选择遵循一个简单规则：谁最有能力承担第一项主张，谁先出现。及时市场事件、具体业务故障或强个人经历都可以开场，但不能为了“像故事”而让一个小例子证明过大的结论。

### 4. 从“修改文字”升级到“修改系统”

这次读者反馈不仅改变了句子，还产生了可复用规则：

- `张力` 不适合随手描述负面业务压力；
- ACTOR 是 framework，不是 lens；
- `Recursive` 应保持为一个单词，与其他 ACTOR 标签同级；
- 自我提升必须说明学习信号、人工参与和写回路径；
- 普通商业概念不应为了显得专业而保留英文；
- 文章不能默认一句一段；
- 一个严肃论证不应同时推销多套执行框架。

但不是每个偏好都应该变成全局硬规则。反馈需要经过四级判断：

| 层级 | 处理方式 | 适用条件 |
|---|---|---|
| 当前句子 | 只改正文 | 偶发措辞或局部事实 |
| 当前文章 | 写入 prose / visual review | 与本文结构或主题有关 |
| 全局工作流 | 写入 strategy 或 skill | 在其他文章中也大概率重复 |
| 自动检查 | 增加测试或 validator | 能稳定识别，且误报成本可控 |

这能避免两个极端：同样的问题反复出现，或者工作流因为一次反馈不断膨胀。

### 5. 从“图片风格统一”升级到“视觉节奏统一”

之前的 liquid glass 和 Fluent 风格本身并不差，问题是它们重复覆盖真实图片以后显得沉重，而且每张图承担相似功能。这次有效的改变不是换了一套滤镜，而是先决定每张图要解释什么，再决定它应该是什么视觉模式。

最终图片分别承担不同任务：

- 封面制造 Token 与运营价值之间的断点；
- FDE 场景展示工程师与运营人员解决真实供应问题；
- 博客工作流图说明任务链如何变成学习系统；
- 咨询/FDE 图解释工作重心和反馈去向，而不是制造输赢；
- ACTOR 图负责方法框架；
- 结尾图负责客户所有权和价值转化。

以后视觉流程继续遵守：

```text
文章论点锁定 -> 每张图的 visual predicate
-> 风格方向菜单 -> Aaron 选择统一 / controlled mix / experimental mix
-> 候选生成 -> 语义审图 -> contact sheet 审节奏
-> WebP -> 正文插入 -> 浏览器实测
```

审图顺序必须是语义正确、构图清楚、文字准确、风格质量。一个漂亮但箭头错误的 ACTOR 图仍然是不合格图片。

## 暴露出来的根本问题

### 下游生产开始得太早

正文还在发生结构性变化时，我们已经生成了图片、音频和视频。结果是每次文章变好，下游资产就变旧。Package validator 能发现过期，但发现发生在成本已经产生以后。

解决办法不是减少迭代，而是引入三个明确的锁：

1. **Argument Lock：**主张、证据、机制、反方和主要框架稳定；通过后才做最终双语 prose 和正式视觉规划。
2. **Article Lock：**英文、中文、标题、章节和图片位置稳定；通过后才生成最终图片、音频和视频。
3. **Package Lock：**博客仓库转换、链接、图片、前端渲染和双语切换全部通过；通过后才允许线上发布。

任何锁之后发生上游修改，都必须明确标记哪些下游资产失效。不能因为文件仍然存在，就把它们当作最新成品。

### “完成”曾经按文件判断，而不是按读者体验判断

旧工作流容易把“Markdown 已生成”“图片已存在”“视频已渲染”当作完成。实际发布中出现过站内链接 404，也出现过本地 Nuxt 旧进程返回 500。它们说明成品必须在最终消费界面验收。

本地发布至少要检查：

- 中英文 URL 都返回正确页面；
- 标题、描述、分类和语言切换正确；
- 封面不在正文重复；
- 所有正文图片在懒加载后具有真实尺寸；
- 站内链接到达已存在的对应语言页面；
- 页面没有横向溢出和控制台错误；
- 本地服务器不是引用旧构建缓存的僵尸进程。

## 下一篇文章的推荐流程

### Phase 0：定义值得写的问题

先回答五个问题，不急着定标题：

1. 最近真正发生了什么变化？
2. 哪类读者正在承受什么压力、误判或机会？
3. 我能提供什么不是来源摘要的解释？
4. 我的权威边界在哪里？
5. 读者读完以后能做出什么不同判断？

Kill criteria 也要提前写清楚：如果没有非显而易见的观点、没有足够一手证据，或者主题超出我的判断边界，就换角度或停止。

### Phase 1：回看自己的长期观点

检查至少三篇旧文，但不强行站内引用。目标是找出：

- 哪个旧观点可以升级；
- 哪个旧例子已经过时；
- 哪个内部链接真正帮助当前读者；
- 当前文章会给个人 canon 增加什么新判断。

内部资料默认用中文写，来源原文和专有名词保留英文，降低 Aaron 审阅成本。

### Phase 2：研究与证据约束

先完成 research dossier，再完成 claim ledger。优先一手来源，并主动寻找反例、失败案例和利益相关方偏见。Vendor announcement 只证明战略和自我描述，不自动证明结果。

研究结束条件不是“资料很多”，而是关键主张已经有足够证据，剩余未知能够被清楚标记。

### Phase 3：论证设计

Argument memo 只解决一件事：文章如何从一个事实走到一个值得争论的判断。确定：

- 一句话 thesis；
- why now；
- core mechanism；
- strongest evidence；
- credible objection；
- one executable framework；
- final implication。

这一步通过 Argument Lock。没有通过，不进入精细 prose 和最终配图。

### Phase 4：结构稿与压缩

先写结构完整的 draft，再做 structural redraft。每个 section 必须承担不同工作：提出因果、展示机制、提供证据、处理反方、给出执行方法或推进结论。

默认目标：英文 1,600-2,600 词，五到七个主要章节，每节两到三个完整段落。短段落只用于真正需要强调的句子，不作为默认排版风格。

Red team 必须尝试删除至少一个段落、一个概念或一个框架，而不只是补充内容。

### Phase 5：双语与文字审校

英文和中文保持同一 claim set，但分别写成自然文章。中文检查普通商业术语、抽象词方向、句子节奏和段落完整性；英文检查 generic AI language、公式化对比和不自然的 headline prose。

Prose pass 不能偷偷新增事实或改变主张。标题、开头和结尾需要单独读一遍，确保三者承诺的是同一篇文章。

这一步通过 Article Lock。Aaron 明确接受正文后，再进入最终图片和媒体生产。

### Phase 6：视觉生产

先呈现 3-5 个适合本文的风格方向，并明确建议统一、controlled mix 还是 experimental mix。Aaron 选择后，再做 prompt pack 和候选图。

正文长不等于图片越多越好。每张图必须能回答“没有这张图，读者少理解了什么”。框架图必须验证顺序、箭头、标签和层级；人物比较图不能用无依据的输赢隐喻。

### Phase 7：本地发布验收

同步到博客仓库时，如果 slug 已存在，保留文章编号和稳定 URL。转换 frontmatter、图片路径和站内链接后，运行 link validator、相关前端测试和本地 dev server。

浏览器同时验收中英文页面、首屏、文章中段图片、底部内容、懒加载、控制台和横向溢出。页面通过以后才达到 Package Lock。

### Phase 8：媒体适配与发布

视频不是朗读博客，音频也不是把 Markdown 直接送入 TTS。先重新定义视频 promise、cold open、story spine 和画面节奏。文章在 Article Lock 后发生任何变化，都要重新审计 script、audio 和 video。

外部推送仍然需要明确授权。发布后在 24 小时和 7 天复盘 qualified reading、保存、回复、订阅和渠道表现，而不只看点击量。

## 人与 AI 的职责边界

AI 可以负责：

- 扩大研究范围；
- 整理一手资料；
- 生成多个论证和视觉候选；
- 找矛盾、重复和缺失；
- 执行双语、图片、链接和包验证；
- 把已经确认的规则自动化。

Aaron 必须保留：

- 选择真正值得写的问题；
- 判断哪些来源可信、哪些边界不能越过；
- 决定 thesis、标题、语气和主要框架；
- 拒绝看起来完整但没有价值的内容；
- 选择视觉方向和最终成品；
- 决定哪些反馈代表长期观点变化。

高参与度不是工作流失败，而是当前高质量生产的组成部分。目标不是假装人已经不需要参与，而是让每次参与留下更清楚的标准，使未来同类判断更快、更稳定。

## Artifact 应该保留多少

### 对严肃文章继续保留

- `editorial-brief.md`
- `research-dossier.md`
- `claim-ledger.md`
- `argument-memo.md`
- `canon-alignment.md`
- `red-team-review.md`
- `prose-polish-review.md`
- `editorial-scorecard.md`
- `postmortem.md`
- visual strategy / manifest / critique / postmortem

这些 artifact 分别关闭了真实风险，不是为了展示流程复杂度。

### 下一篇可以实验合并

- `content-plan.md` 与 `plan.md` 是否可以合并；
- `memory-reflection.md` 与 `canon-alignment.md` 是否存在重复；
- 社交文案是否应该在文章发布前全部生成，还是只保留首发渠道需要的版本。

合并标准只有一个：减少维护成本，但不能让证据、论证、语言或发布责任重新混在一起。

## Definition Of Done

一篇严肃文章只有同时满足下面六项才算完成：

- **Argument：**有明确、可争论、非摘要式的主张；
- **Evidence：**核心事实有一手来源，推断和判断被正确标记；
- **Language：**中英文自然、准确，段落与标题服务论证；
- **Visual：**每张图增加理解，整体有节奏，语义没有错误；
- **Package：**最终博客页面、图片、链接、语言切换和前端运行全部通过；
- **Learning：**至少一条反馈被归纳为下一篇可复用的规则或实验。

## 下一篇的两个实验

1. 在不降低质量的前提下，合并 `content-plan.md` 与 `plan.md`，观察是否减少重复审阅。
2. 只在 Article Lock 后生成最终图片和媒体，记录由正文返工导致的废弃资产是否明显减少。

视觉上可以继续测试一张受控的深色或纪实背景图片，但前提是它服务文章节奏，而不是为了证明新工作流能够生成更多风格。

这次最大的收获不是得到了一篇更好的 FDE 文章，而是更清楚地知道质量从哪里产生：不是一次生成，而是一连串有边界的判断，以及这些判断能否成为下一次运行的起点。
