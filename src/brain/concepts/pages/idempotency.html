Idempotency

同一个业务意图即使被重复送达，系统也只让它产生一次预期的业务效果。

A small equation, a large system contract
f(f(x)) = f(x)

真正的问题不是“服务器能不能认出重复请求”，而是：服务端已经成功、响应却消失时，客户端能不能安全地再试一次？

01 · Unknown outcome

扣款已经成功，响应却消失了。

客户端没有成功或失败的证据。切换两种模式并逐步重试：没有身份时，每次 POST 都像一条新命令；有稳定幂等键时，多次尝试仍然表达同一个业务意图。

02 · Key contract

难处理的情况都有规则，key 才真正有用。

一个 header 不是保证。完整协议要定义意图身份、参数绑定、原子执行权、并发重复以及过期窗口。

Request相同调用者 · 相同 key · 相同参数
Decision查到完成记录，不再执行业务效果。
→
Result重放第一次结果，或返回语义等价结果。

◆ 重试靠业务意图识别，不靠它是第几次到达。

03 · Operation judgment

先问：这个操作能否自己收敛？

自然幂等比维护账本更简单。“设置成某状态”常会收敛；“再增加一次”和外部副作用需要更明确的身份协议。

SET status = PAID

自然幂等

每次重复都收敛到同一个被明确命名的状态。

DELETE document 42

自然幂等

响应可以不同，但资源最终始终不存在。

POST create payment

需要意图 key

如果重试没有共享身份，每次创建都会成为一笔新扣款。

balance = balance + 10

非自然幂等

每次执行都会再次改变当前状态。

最小
记忆卡

  1. 幂等性处理的是结果未知：允许重试，不允许同一意图重复产生业务效果。
  2. 一个意图一个 key；重试复用它；新意图换新 key。
  3. 相同 key + 不同参数必须拒绝；并发和过期行为必须写进契约。
  4. 最危险的 bug 位于副作用与幂等记录之间的非原子缝隙。
  5. 幂等性不是 safe、去重、重试或 exactly-once 的同义词。
