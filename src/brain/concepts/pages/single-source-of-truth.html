Authority · Lineage · Recovery

Single Source
of Truth

Single Source of Truth（SSOT，单一事实来源）不是只保存一份数据，而是让同一个事实只有一个被授权的裁决者；其余副本都知道自己从哪里来、多久会过期、怎样重建。

一个事实可以有很多副本，
但只能有一个明确的修正入口。

01 · Drift lab

亲手把一条发布链变成两个真相。

切换三个场景，观察 source、public package、blog copy 与 production 的角色怎样变化。文件数量不决定健康程度，权威方向才决定。

一个权威源，可重复生成多个投影

源头前进到 version 4；所有下游表示沿一个方向重新生成，并记录同一个 source version。

Healthy

物理副本可以很多，修正路径只有一条，而且没有歧义。

先恢复权威，再恢复一致。

02 · Authority contract

四个问题都有答案，SSOT 才真正存在。

把一个文件叫作 canonical 只是承诺。周围的流程必须让所有权、方向、新鲜度和恢复路径都可以执行。

01authority

谁可以发起一次修正？

为这类有边界的事实，明确拥有者与唯一被接受的写入路径。

02direction

更新朝哪个方向流动？

除非领域真的需要多个写入者，否则优先使用单向、可重复的生成。

03freshness

副本允许落后多久？

声明 source version、刷新触发条件和业务可以接受的延迟。

04recovery

删除它以后怎样重建？

每个派生物都需要确定、可重复而且经过验证的恢复路径。

03 · Boundary check

“单一”最容易制造三种误会。

SSOT 集中的是某一类事实的裁决权，而不是把所有系统、数据与副本强行塞进一个地方。

物理存储

误区：一个源，就只能有一份副本。

Replica、缓存、报表和搜索索引都应该存在。它们可以落后、失效与重建，只是不能悄悄获得定义事实的权力。

系统设计

误区：一个源，就是一个巨大数据库。

Order Service 可以拥有订单，Identity Service 可以拥有法定姓名。权威应该按 domain boundary（领域边界）切分。

数据质量

误区：权威源一定自动正确。

它仍然可能出错。SSOT 的价值是让所有人知道修正落在哪里，并能从那里重新生成下游。

04 · Concept neighborhood

邻居是在拥有、追踪，还是投影这个事实？

知道每个术语属于哪一种关系，比把它们当成 SSOT 的同义词更有用。

Canonical Source

authority mechanism

Canonical Source · 规范源

被正式选为某类事实权威表示的具体文件、存储或日志。

Lineage

provenance

Data Lineage · 数据血缘

记录数据从哪里来、经过哪些转换，以及由哪个 source version 产生。

Materialized View

derived projection

Materialized View · 物化视图

为查询保存的派生表示；它可以落后，并且应该能从源头重建。

Event Sourcing

implementation

Event Sourcing · 事件溯源

把按顺序排列的事件日志作为权威记录，通过 replay 构建当前状态。

CQRS

architecture pattern

Command Query Responsibility Segregation · 命令查询职责分离

把权威命令与读优化投影分开，同时显式承担同步延迟。

SVOT

consumption agreement

Single Version of Truth · 单一版本的真相

强调消费者对统一定义或结果达成一致；SSOT 更强调权威从哪里产生。

Takeaway

先在源头修，再重新生成下游。

最值得记住的不是缩写 SSOT，而是一个判断动作：每当你不知道“应该改哪个文件、信哪个数字、从哪边同步”时，先停止复制，明确这个事实的权威、方向、版本与恢复路径。
