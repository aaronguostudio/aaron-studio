01 · The intuition

它不是“没有控制”，而是把控制放在最后一刻。

两个请求可以同时读取同一份数据、各自完成工作。真正提交时，系统验证：“我开始工作时看到的版本，现在还成立吗？”

One-sentence definition

读取时带走一个版本标记；写入时只有版本仍未改变才允许提交，否则把它当成一次显式冲突。

关键：OCC（Optimistic Concurrency Control，乐观并发控制）的价值不是让冲突消失，而是阻止 Bob 基于旧 v7 静默覆盖 Alice 已提交的 v8。冲突被看见以后，系统才能选择拒绝、重试或合并。

03 · Under the hood

三个阶段：Read、Work、Validate + Write。

最重要的设计点，是把验证和写入做成同一个不可分割的原子动作。

01

Read

读取业务数据，同时带走 version、ETag（Entity Tag，实体标签）或内容哈希。

02

Work

本地编辑或计算；这段时间不长期占有排他锁。

03

Validate + Write

版本仍相同才写入并递增；不同则返回明确冲突。

UPDATE documents
SET title = 'Q3 Growth Plan',
    version = version + 1
WHERE id = 42
  AND version = 7;

-- affected rows = 1  → success, now v8
-- affected rows = 0  → conflict: v7 is stale

不要拆开：如果先 SELECT 检查 version，过一会儿再单独 UPDATE，两步之间仍可能有人插入修改。这是 TOCTOU（Time of Check to Time of Use，检查时刻到使用时刻）race。条件必须进入原子 UPDATE、transaction validation 或 CAS（Compare-and-Swap，比较并交换）。

04 · Strategy switch

乐观与悲观，赌的是不同成本。

两种方法都在保护正确性。区别在于：是接受偶尔返工，还是让所有人预先等待？点击切换观察工作方式。

大家同时工作，提交时验票

无冲突时几乎没人等待；冲突时，至少一方的工作需要重试、合并或放弃。

Best when冲突少
Main risk重试风暴
乐观成本 ≈ 验证成本 + 冲突概率 × 重做成本
悲观成本 ≈ 加锁成本 + 等待成本 + 死锁 / 超时处理成本
05 · Concept neighborhood

最容易误会的，是那些“看起来很像”的邻居。

把策略、机制、存储模型和用户体验分开，很多并发术语会立刻清晰。

MVCC STORAGE MODEL

Multi-Version Concurrency Control · 多版本并发控制

保存多个版本，让读者看到一致快照，减少读写互相阻塞。它主要回答“读哪个版本”。

CAS ATOMIC PRIMITIVE

Compare-and-Swap / Compare-and-Set · 比较并交换

只有当前值仍等于 expected 才替换。它是一种可实现 OCC 的原子积木。

Isolation TRANSACTION SEMANTICS

Transaction Isolation · 事务隔离

决定并发事务能观察什么。用了 transaction 不自动等于不会 lost update。

Optimistic UI USER EXPERIENCE

Optimistic User Interface · 乐观式界面

界面先展示成功、失败再回滚；它优化感知速度，不负责并发正确性。

OT / CRDT MERGE MODEL

Operational Transformation / Conflict-free Replicated Data Type · 操作转换 / 无冲突复制数据类型

定义并发操作如何转换或收敛，常用于实时协作，不只是“发现冲突就拒绝”。

Lease TIME-BOUND OWNERSHIP

Lease · 租约

会过期的临时所有权；比永久锁更能容忍崩溃，但要处理续租、时钟与 fencing。

一句辨别：OCC（Optimistic Concurrency Control）是“什么时候检查写入前提”的策略；MVCC（Multi-Version Concurrency Control）是“怎样保存并呈现版本”；CAS（Compare-and-Swap）是“怎样原子比较并替换”的机制。三者可以同时存在。
06 · Implementation recipes

Version 不是装饰，冲突也不是异常边角。

一个可用的 OCC 协议必须覆盖完整闭环：把版本交出去、原子验证、识别失败，然后设计冲突体验。

HTTP（Hypertext Transfer Protocol）：ETag（Entity Tag）+ If-Match

# 读取时，服务端交付版本
GET /documents/42
→ ETag: "v7"

# 更新时，客户端声明自己的前提
PUT /documents/42
If-Match: "v7"

# 如果资源已是 v8
→ 412 Precondition Failed

冲突之后的四种出口

策略适合主要代价
拒绝并刷新人工编辑、后台管理用户需要重新确认
自动重读 + 重试短小、确定性、幂等操作高争用会 retry storm
字段或语义合并独立字段、协作内容合并规则本身复杂
串行化热点库存、竞拍、单一热点吞吐受队列 / 锁限制

自动重试的前置条件：幂等性、指数退避 + jitter、重试上限。一次操作如果包含扣款、发邮件或第三方 API，盲目重跑可能比原始冲突更糟。

08 · Pocket memory

最后，只记住这五件事。

  • 01不预先阻塞，提交时验证前提。这是 OCC 的核心。
  • 02检查与写入必须原子化；version、ETag、CAS 都只是载体。
  • 03冲突是协议的一部分，必须设计拒绝、重试或合并。
  • 04低争用、长交互偏乐观；高争用、短临界区常偏悲观或串行化。
  • 05MVCC、事务、隔离、CAS 与 OCC 有关，但都不等于 OCC。
