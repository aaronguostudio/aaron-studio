# Blog Reinforcement Loop v0 Design

## Purpose

Turn Aaron's blog-growth system from a reporting layer into a working self-improvement loop.

Version 0 has one concrete promise:

```text
Every published article gets a 7-day postmortem, and that postmortem influences the next article brief.
```

This is not a generic reinforcement learning platform. It is a practical product loop for Aaron's blog: publish, distribute, measure, review, learn, and feed those lessons back into the next writing and publishing workflow.

## Current Context

The existing blog-growth system already has the first measurement spine:

- `growth_content_items` stores scanned blog content.
- `growth_web_path_map` maps blog paths to content items.
- `growth_metric_snapshots` stores normalized daily metrics.
- `growth_metric_catalog` defines canonical metrics and reward weights.
- `growth_content_daily_scorecard`, `growth_content_lifetime_scorecard`, `growth_content_7d_postmortem`, `growth_weekly_scorecard`, and `growth_topic_scorecard` provide analysis views.
- `scripts/blog-growth.mjs` supports content scan, Rybbit ingestion, post-publish ingestion, and UTM URL generation.
- `blog-production`, `blog-write`, and `publish-to-blog` already mention growth workflow expectations.

The system is still missing the actual learning loop:

- `growth_channel_posts` has no reliable post-publish registration flow.
- `growth_ai_reviews` is empty and not yet written by a postmortem command.
- YouTube Analytics can be queried directly, but is not ingested into `growth_metric_snapshots`.
- LinkedIn analytics needs a manual import path before API read permissions are solved.
- `published_at` currently stores display dates such as `15th Jun 2026`, so SQLite `date(published_at)` cannot power 7-day postmortem windows.
- Reward weights are duplicated in the scorecard view and should become version-aware before the score is treated as a stable learning target.

## Design Principles

1. Keep the loop small enough to run after every article.
2. Prefer explicit workflow artifacts over hidden memory.
3. Store summaries, evidence, and recommendations, not hidden chain-of-thought.
4. Treat metrics as directional until source coverage improves.
5. Make every automated recommendation traceable to a content item, metric window, and review record.
6. Use existing `scripts/blog-growth` patterns instead of adding a new app.

## Selected Approach

Use the existing Turso schema and CLI as the system boundary, and add the smallest set of commands needed to close the loop:

```text
post-publish registration
  -> metrics ingestion
  -> 7-day postmortem generation
  -> growth_ai_reviews writeback
  -> next-brief memory read
```

This approach keeps the work inside Aaron Studio. In v0, `scripts/blog-growth` owns the data loop and `blog-production` reads its outputs before the next article workflow.

### Alternatives Considered

#### A. Build a full dashboard first

This would make the system easier to inspect, but it would not make the loop learn faster. The current bottleneck is not visualization; it is writeback and reuse.

#### B. Build a general personal reinforcement platform

This matches the long-term vision, but it is too large for the first implementation. The blog gives a better test bed because it already has repeated tasks, public outputs, distribution channels, and measurable outcomes.

#### C. Start with manual reviews only

This is low risk, but it would keep the learning loop outside the product system. Version 0 should already write to `growth_ai_reviews`, even if the first postmortem text is generated by an agent command.

## Version 0 Scope

### In Scope

1. Normalize publication dates during content ingestion.
2. Register distribution artifacts in `growth_channel_posts`.
3. Ingest YouTube Analytics metrics for registered YouTube channel posts.
4. Add a manual LinkedIn metrics import path.
5. Generate and store 7-day postmortems in `growth_ai_reviews`.
6. Add a next-brief memory read that summarizes recent winning and losing patterns.
7. Update blog-production guidance so every future publish runs the reinforcement loop.

### Out Of Scope

1. A public dashboard UI.
2. Fully automated LinkedIn API analytics.
3. Newsletter/Beehiiv subscriber attribution.
4. Model training or parameter-level reinforcement learning.
5. Multi-armed bandit allocation.
6. Automated topic selection without Aaron's approval.
7. Rewriting all historical posts.

## Data Model Changes

### Publication Date Normalization

`growth_content_items.published_at` should store ISO dates: `YYYY-MM-DD`.

The ingestion code should parse common blog frontmatter dates, including:

- `2026-06-15`
- `15th Jun 2026`
- `15 Jun 2026`
- `June 15, 2026`

If a date cannot be parsed, ingestion should keep the raw value inside `raw_metadata_json` and store `published_at = NULL`.

Acceptance check:

```sql
SELECT COUNT(*) AS unparseable_count
FROM growth_content_items
WHERE published_at IS NOT NULL
  AND date(published_at) IS NULL;
```

The expected v0 result is `0`.

### Channel Post Registration

`growth_channel_posts` should become the bridge between content and distribution.

Version 0 should support a local manifest convention in each article workflow directory:

```text
distribution.json
```

Example:

```json
{
  "slug": "fable-5-managing-ai-autonomy",
  "posts": [
    {
      "channel": "linkedin",
      "channel_post_id": "manual-or-url-id",
      "channel_url": "https://www.linkedin.com/feed/update/...",
      "title": "Fable 5 Changed the Unit of AI Work",
      "language": "en",
      "post_type": "brief",
      "hook_text": "Fable 5 changed what I think an AI work unit is.",
      "cta_type": "blog_click",
      "published_at": "2026-06-15"
    },
    {
      "channel": "youtube",
      "channel_post_id": "jPHR-73HJa8",
      "channel_url": "https://www.youtube.com/watch?v=jPHR-73HJa8",
      "title": "Fable 5 Changed the Unit of AI Work",
      "language": "en",
      "post_type": "video",
      "cta_type": "blog_click",
      "published_at": "2026-06-15"
    }
  ]
}
```

Add a CLI command:

```bash
node scripts/blog-growth.mjs register-channel-posts --file <path>
```

It should upsert rows into `growth_channel_posts` by `(channel, channel_post_id)` when `channel_post_id` exists, otherwise by `(content_item_id, channel, channel_url)`.

### Reward Score Versioning

Version 0 should add a lightweight score version marker without refactoring the whole scorecard.

Add table:

```sql
CREATE TABLE IF NOT EXISTS growth_reward_versions (
  version TEXT PRIMARY KEY,
  description TEXT NOT NULL,
  metric_weights_json TEXT NOT NULL,
  active_from TEXT NOT NULL,
  active_to TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Add `reward_version` to `growth_ai_reviews.raw_context_json` for generated postmortems.

Do not rebuild every scorecard view in v0. The purpose is to make review outputs interpretable when weights change later.

## CLI Commands

### 1. `normalize-content-dates`

Purpose: repair current catalog rows after the parser is updated.

Command:

```bash
node scripts/blog-growth.mjs normalize-content-dates --dry-run
node scripts/blog-growth.mjs normalize-content-dates
```

Behavior:

- Re-scan content markdown.
- Parse publication dates to ISO.
- Upsert `growth_content_items`.
- Print counts for total, normalized, null, and unparseable.

### 2. `register-channel-posts`

Purpose: link distribution artifacts to content items.

Command:

```bash
node scripts/blog-growth.mjs register-channel-posts --file <distribution.json>
```

Behavior:

- Validate slug exists in `growth_content_items`.
- Validate allowed channel values.
- Normalize `published_at`.
- Upsert `growth_channel_posts`.
- Print inserted, updated, skipped, and warning counts.

### 3. `ingest-youtube`

Purpose: write YouTube Analytics metrics into `growth_metric_snapshots`.

Command:

```bash
node scripts/blog-growth.mjs ingest-youtube \
  --start 2026-06-15 \
  --end 2026-06-22 \
  --slugs fable-5-managing-ai-autonomy
```

Behavior:

- Read registered YouTube channel posts for selected slugs or date window.
- Use the existing YouTube OAuth helper to get a valid access token.
- Query YouTube Analytics by video and day where possible.
- Write canonical metrics:
  - `youtube_views`
  - `youtube_watch_minutes`
  - `youtube_average_view_duration_seconds`
  - `youtube_subscribers_gained`
  - `youtube_likes`
  - `youtube_comments`
- If `youtube_impressions` or CTR are rejected by the API, record a warning and skip those metrics.
- Use `entity_type = 'channel_post'` and `entity_id = growth_channel_posts.id`.

### 4. `import-linkedin`

Purpose: provide a manual analytics fallback while LinkedIn API access is restricted.

Command:

```bash
node scripts/blog-growth.mjs import-linkedin --file <linkedin-metrics.csv>
```

Minimum CSV columns:

```text
slug,channel_post_id,metric_date,impressions,members_reached,reactions,comments,reshares,link_clicks,followers_gained
```

Behavior:

- Match each row to `growth_channel_posts` by `channel = 'linkedin'` and `channel_post_id`.
- If no channel post exists, fail with a clear message and suggest `register-channel-posts`.
- Write available values into canonical LinkedIn metric names.
- Ignore blank metric cells.

### 5. `postmortem`

Purpose: generate and store a postmortem for one content item.

Command:

```bash
node scripts/blog-growth.mjs postmortem \
  --slug fable-5-managing-ai-autonomy \
  --window 7d
```

Supported windows:

- `24h`
- `7d`
- `30d`

Behavior:

- Resolve content item by slug.
- Calculate the metric window from ISO `published_at`.
- Query content scorecard rows and linked channel metrics.
- Produce a compact review payload:
  - summary
  - expected outcome
  - actual outcome
  - strongest signals
  - weakest signals
  - distribution notes
  - likely explanation
  - recommended actions
  - reusable writing patterns
  - reusable distribution patterns
  - next experiment suggestion
- Insert into `growth_ai_reviews` with:
  - `review_type = postmortem_7d`, `postmortem_24h`, or `postmortem_30d`
  - `content_item_id`
  - `period_start`
  - `period_end`
  - `summary`
  - `insights_json`
  - `recommended_actions_json`
  - `raw_context_json`

The command may initially generate a structured markdown/JSON draft for the agent to write back. It does not need a model API inside the CLI in v0 if Codex is the agent that writes the final review.

### 6. `next-brief-context`

Purpose: feed recent learnings back into `blog-outline`, `blog-write`, and `blog-production`.

Command:

```bash
node scripts/blog-growth.mjs next-brief-context --limit 5
```

Behavior:

- Read recent `growth_ai_reviews`.
- Read top lifetime and recent scorecards.
- Return a compact JSON/markdown context:
  - winning topics
  - losing or weak patterns
  - hooks to reuse
  - hooks to avoid
  - distribution actions to repeat
  - open measurement caveats
  - recommended next experiment

This command is the key bridge from measurement to writing.

## Blog Workflow Integration

### `publish-to-blog`

After a successful publish:

1. Run `ingest-after-publish`.
2. Generate or update `distribution.json` with known channel artifacts.
3. Run `register-channel-posts` when channel metadata exists.
4. Remind the agent to schedule or run 24h and 7d postmortems.

### `blog-production`

At the start of a new blog workflow:

1. Run or read `next-brief-context`.
2. Include the top lessons in the topic and outline step.
3. Require the article brief to declare:
   - hypothesis
   - target audience
   - expected distribution channel
   - success metric
   - likely reusable pattern

At the end of a publish workflow:

1. Ensure UTM URLs were used.
2. Ensure `distribution.json` exists or explain why no channel artifacts exist yet.
3. Ensure the postmortem plan is recorded.

### `blog-write`

During drafting, use recent lessons as a quality gate:

- Does this post reuse a proven pattern intentionally?
- Does it avoid a pattern that recent postmortems flagged as weak?
- Is the thesis measurable enough for a 7-day postmortem?

## Postmortem Review Shape

Store compact JSON in `growth_ai_reviews.insights_json`:

```json
[
  {
    "type": "content_pattern",
    "label": "practical_ai_native_explainer",
    "evidence": "High score and strong reach from concrete AI explainer posts.",
    "confidence": "medium"
  },
  {
    "type": "distribution_gap",
    "label": "direct_traffic_dominates",
    "evidence": "Rybbit baseline shows direct traffic dominates while attributed social is small.",
    "confidence": "high"
  }
]
```

Store actions in `recommended_actions_json`:

```json
[
  {
    "action": "Write another practical AI-native explainer with a concrete metaphor.",
    "owner": "blog-production",
    "priority": "high"
  },
  {
    "action": "Register LinkedIn and YouTube channel posts before judging distribution.",
    "owner": "blog-growth",
    "priority": "high"
  }
]
```

Store source context in `raw_context_json`:

```json
{
  "reward_version": "v0.1",
  "metric_window": "7d",
  "metric_date_start": "2026-06-15",
  "metric_date_end": "2026-06-22",
  "sources": ["growth_content_daily_scorecard", "growth_channel_metric_rollup"],
  "known_gaps": ["linkedin_manual_import_missing"]
}
```

## Error Handling

### Date Parsing

If a content date cannot be normalized, keep the content item but exclude it from postmortem windows. The CLI should show the source file and raw date string.

### Channel Registration

If a channel post references a slug that is not in `growth_content_items`, fail before writing any rows for that file.

### YouTube API

If the token is expired, attempt refresh through the existing helper. If the API rejects a metric such as impressions, write a warning and continue with supported metrics.

### LinkedIn Import

If a CSV row cannot be matched to a registered channel post, fail that row and report it. Do not create implicit LinkedIn channel posts from metrics imports.

### Postmortem Generation

If the selected window has no metrics, still write a postmortem only when the missing data itself is the finding. Mark the review with a low-confidence insight and a recommended instrumentation action.

## Success Criteria

Version 0 is done when:

1. `growth_content_7d_postmortem` returns rows for recent posts with ISO publication dates.
2. At least one blog post has LinkedIn and YouTube channel objects in `growth_channel_posts`.
3. YouTube metrics for a registered video are written into `growth_metric_snapshots`.
4. LinkedIn metrics can be imported from a manual CSV.
5. At least one `postmortem_7d` row is written to `growth_ai_reviews`.
6. `next-brief-context` returns lessons from `growth_ai_reviews`.
7. `blog-production` requires reading those lessons before the next outline/write cycle.

## Implementation Order

1. Date normalization and tests.
2. Channel post registration and tests.
3. YouTube ingestion and tests.
4. LinkedIn manual import and tests.
5. Postmortem context/query builder and writeback.
6. Next-brief context command.
7. Skill updates for `blog-production`, `blog-write`, and `publish-to-blog`.
8. One end-to-end smoke run on an existing post.

## Deferred Decisions With v0 Defaults

1. `distribution.json` should live in article workflow directories for v0 because those already hold publishing artifacts. A future version may mirror the manifest into the blog repo.
2. Codex should generate postmortem text in v0 while the CLI handles query and writeback. This avoids adding another model dependency inside `scripts/blog-growth`.
3. English and Chinese translations should remain separate content identities in v0. Postmortems should note translation effects explicitly when they matter.
